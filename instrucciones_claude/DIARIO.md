# DIARIO — Bitácora compartida de Claudes

> **Qué es esto.** Una bitácora append-only donde cada Claude (el de Santi y el del Fran) deja registro de **qué hizo**, **por qué**, **qué errores se encontraron**, y **qué debería saber el otro**. Sirve para que la próxima sesión (cualquiera de los dos) arranque con contexto real de lo que ya pasó.
>
> **Regla de uso para el Claude que esté activo:**
> 1. Al **arrancar sesión**: leer las últimas ~10 entradas (las más recientes arriba).
> 2. Al **cerrar una tarea no trivial** (feature, bug-fix, decisión arquitectónica, cambio en infra, fix de build, etc.): agregar una entrada siguiendo el formato de abajo.
> 3. **No editar entradas viejas.** Si algo cambió, agregar entrada nueva con "corrección" o "actualización".
> 4. Ordenar por fecha **descendente** (lo más nuevo arriba).
>
> **No es para:** changelogs de usuario, release notes, ni PR descriptions. Eso va a otros archivos.
>
> **Cuándo archivar.** Cuando el archivo supere ~300 entradas, moverlas a `DIARIO-archivo-YYYY-Qn.md` y dejar este vacío.

---

## Formato de entrada

```markdown
## YYYY-MM-DD — <autor: Santi|Fran> — <área: backend|frontend|infra|db|auth|...>
**Qué:** <qué cambió en 1-2 líneas>
**Por qué:** <motivación; requisito / bug / decisión>
**Problemas:** <errores que aparecieron y cómo se resolvieron; omitir si no hubo>
**Impacto para el otro:** <qué necesita saber la otra persona; omitir si no aplica>
**Refs:** <commits, archivos clave, PRs; opcional>
```

**Por qué:** investigación web de la API Moodle Web Services confirma que las funciones que necesitamos (`core_user_create_users`, `core_user_get_users_by_field`, `enrol_manual_enrol_users`, `enrol_manual_unenrol_users`) son estables desde Moodle 2.0/3.0 y siguen vivas en 4.1, 4.3, 4.5 (LTS) y 5.0. Moodle 5.0 hizo limpieza de deprecadas pre-4.0 (`core_courseformat_create_module` entre otras) pero nuestra superficie API no está afectada. Contrato REST único (`POST /webservice/rest/server.php?wstoken=...&wsfunction=...&moodlewsrestformat=json`) no varía entre versiones. Conclusión: se puede programar el cliente genérico ya contra el contrato estándar; los únicos bloqueos son operativos (admin Moodle debe habilitar REST, crear servicio externo con esas funciones, emitir token, compartir IDs de cursos y el roleid de "student" si es custom).
**Problemas:** ninguno. Decisiones:
  - IMEDBA es source of truth. `courses.moodle_course_id` y `students.moodle_user_id` son INTEGER nullable, no PKs compartidas. Cursos/alumnos pueden existir sin contraparte en Moodle.
  - Feature flag `MOODLE_ENABLED`: permite mergear Fase 7 a prod y deployar sin Moodle listo — los hooks son no-op silenciosos mientras la flag esté en false.
  - `MOODLE_DEFAULT_STUDENT_ROLE_ID` como env var (default 5 = Moodle vanilla) porque roles en Moodle pueden ser customs con IDs arbitrarios; hardcodear es un foot-gun de seguridad (puede inscribir alumnos como teachers si el admin renumeró).
  - Log de sync: reutilizar `notifications` con `notification_type=MOODLE_SYNC` y `related_entity_type=ENROLLMENT` para no crear tabla nueva (acorde con la filosofía "entidades unificadas" del ERD). Retry ya existe en NotificationService.
**Impacto para el otro:** cuando empiece la Fase 7 real del código, aparecen env vars nuevas: `MOODLE_URL`, `MOODLE_TOKEN`, `MOODLE_DEFAULT_STUDENT_ROLE_ID`, `MOODLE_ENABLED`. En dev pueden quedar en vacío/false. En prod se pobrarán cuando el admin de Moodle entregue token y URL. No hay cambios en endpoints del SPA por ahora.
**Refs:** `instrucciones_claude/05-moodle-integration-spec.md` (actualizado), `instrucciones_claude/ESTADO.md` (próximo paso corregido).
